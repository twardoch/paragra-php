# Style Guide

Conventions for code, documentation, and API design in `paragra-php`.

---

## PHP code style

### Formatting

- **PHP-CS-Fixer** is authoritative. Run `composer lint` before committing.
- `declare(strict_types=1);` at the top of every file, after the opening `<?php` tag.
- `// this_file: paragra-php/path/to/file.php` on the line after `declare`.
- No closing `?>` tag.
- Blank line after `namespace` declaration.
- Import-group order: PHP built-ins, then Composer packages, then internal namespaces — each group separated by a blank line. Within a group, alphabetical.
- `use function` and `use const` blocks after `use class` blocks.

### Naming

| Element | Convention | Example |
|---|---|---|
| Class | `PascalCase` | `PineconeVectorStore` |
| Interface | `PascalCase` + `Interface` suffix | `VectorStoreInterface` |
| Trait | `PascalCase` + `Trait` suffix | `HasRetryTrait` |
| Method | `camelCase` | `getDefaultNamespace()` |
| Property | `camelCase` | `$apiKey` |
| Constant | `SCREAMING_SNAKE_CASE` | `DEFAULT_FAMILY_POLICIES` |
| Test method | `snake_case`, descriptive | `test_upsert_sends_vectors_and_returns_counts()` |
| Config key | `snake_case` | `api_key`, `priority_pools` |
| Provider slug | `snake_case`, lowercase | `gemini_file_search`, `openai` |

### Type system

- All public method parameters and return types must be declared (no mixed except when unavoidable).
- Use `array<K, V>` generics in PHPDoc when the array shape is not a fixed tuple.
- Use `list<T>` (not `array<int, T>`) for ordered sequences with no gaps.
- Use `array{key: Type, ...}` PHPDoc shapes for fixed-key arrays.
- Prefer `readonly` constructor-promoted properties over setters.
- `final` by default; lift it only with an explicit justification comment.

### Exceptions

- Throw `\InvalidArgumentException` for bad caller input (wrong type, empty required value, etc.).
- Throw `\RuntimeException` for infrastructure failures (HTTP error, exhausted pool, etc.).
- Never swallow exceptions silently; always log or rethrow with context.
- Provider-specific exceptions must implement or extend a common base so callers can catch a single type.

### PHPDoc

Every public method must have a docblock with:

- A one-line summary (imperative mood: "Return…", "Build…", "Execute…").
- `@param` for every parameter that benefits from a description or complex type.
- `@return` with the full type shape if the return type is `array` or `mixed`.
- `@throws` for every exception that can propagate to the caller.

Private and protected methods may omit the docblock if the signature is self-explanatory.

## Test style

### File layout

- One test class per source class, mirroring the `src/` directory structure under `tests/`.
- Test class name: source class name + `Test` suffix, same namespace with `\Tests\` inserted.
- File header: `// this_file: paragra-php/tests/Path/To/ClassTest.php`.

### Test method names

Use the pattern:

```
test_{method_when_condition_then_result}
```

Examples:
- `test_retrieve_when_primary_pool_fails_then_uses_next_pool()`
- `test_upsert_rejects_empty_records()`
- `test_from_config_when_priority_pools_missing_then_throws()`

### Assertions

- Use `self::assertSame()` for scalar equality (strict); `self::assertEquals()` only when type coercion is intentional.
- Include a message as the last argument when the failure would be ambiguous.
- Assert the fewest things necessary to prove the behaviour under test.

### Mocking

- Use `GuzzleHttp\Handler\MockHandler` + `HandlerStack` to mock HTTP calls — never make live network calls in the test suite.
- Use PHPUnit's `createMock()` for interface stubs.
- Prefer concrete `StubXxx` classes over anonymous mocks when the stub is reused across tests.
- Do **not** call `ReflectionMethod::setAccessible(true)` — it has been a no-op since PHP 8.1.

### Coverage target

≥ 90 % line coverage on `src/`. Run with `composer test` (coverage report is not generated by default; enable it locally with `--coverage-text` or `--coverage-html`).

## Documentation style

### Voice and tense

- Use second person ("You can…", "Pass…") in guides and how-tos.
- Use third person ("The method returns…", "ParaGra throws…") in reference pages.
- Present tense throughout.

### Code examples

- Every code example must be runnable as-is (substitute `$_ENV['KEY']` for real keys).
- PHP code blocks use the ` ```php ` fence.
- Console commands use the ` ```bash ` fence.
- Keep examples short — show the minimum necessary to illustrate the point.

### Tables

Use Markdown pipe tables for all matrices and option references. Column alignment:

```
| Left | Center | Right |
|---|:---:|---:|
```

### File metadata

Every Markdown file in `src-docs/` must begin with YAML front matter containing `this_file`:

```yaml
---
this_file: paragra-php/src-docs/path/to/file.md
---
```

### Zensical nav

When adding a new page:
1. Create the `.md` file in the appropriate `src-docs/` subdirectory.
2. Add the entry to the `nav` array in `zensical.toml`.
3. Rebuild with `uvx zensical build --clean` and commit `docs/` alongside the source change.

---

## Git and release conventions

- Commit messages: imperative mood, under 72 characters, no trailing period.
  - `Add PineconeVectorStore mock tests` ✓
  - `Added tests for Pinecone.` ✗
- One logical change per commit.
- Tag format: `v{MAJOR}.{MINOR}.{PATCH}` (SemVer); use `uvx gitnextver@latest .` to bump.
- Do not force-push to `main`/`master`.
